## Problem Statement

회사 사람들끼리 즐겁게 쓸 수 있는 사내 게임 웹 허브. 회원가입 없이 닉네임만으로 즉시 입장 가능하되, 회사 도메인(`@trizn.kr`)으로 가입한 유저는 기록이 영구 저장됨. 멀티/싱글 게임 다양하게, 채팅·친구 기능 포함. 시작은 재미용, 추후 사내 도구로 확장 가능한 구조.

## What Makes This Cool

- **Jstris 수준의 테트리스 사내 클론** — 핵심 끌림. 동료들 화면 보면서 키 누름.
- **Elixir + Phoenix BEAM** — 회사에서 잘 안 쓰는 색다른 스택. 실시간 멀티플레이어 게임에 미친듯이 적합.
- **게스트 모드** — 가입 강제 없음. 누구나 30초 안에 들어와서 한 판.
- **회사 도메인 락** — `@trizn.kr` SMTP 인증으로 외부인 차단. 내부 클럽.
- **Boil the Lake** — 7게임 일괄 출시. AI로 모듈러 인터페이스 짜면 가능.

## Constraints

- 사내 인원 소수 (수십명 추정)
- 비상업, 사이드 프로젝트
- Docker 단일 컴포즈로 배포 (Vercel FE-only는 포기, DB 필요)
- 메인테이너 1명 (사용자 본인)
- 모듈러: 게임 추가가 폴더 하나로 끝나야 함

## Premises

1. **Docker + MySQL + MongoDB 풀스택** — Vercel FE-only는 DB 요구사항과 충돌, Docker로 결론. (사용자 동의)
2. **두 트랙 인증** — 게스트(닉네임만, 휘발성 세션, 기록 제한) + 등록자(`@trizn.kr` 메일 인증, 영구 기록). (사용자 명시)
3. **`@trizn.kr` 도메인 화이트리스트** — 가입 시 메일 도메인 정규식 락. **메일 인증 발송 없음** (2026-04-26 결정). 대신 admin 페이지에서 사용자 정보/승인 관리.
4. **모든 유저 노출 + 추천 친구 10명 + 더보기** — 회사 작아서 OK. 친구는 양방향 수락 (`pending → accepted`).
5. **모듈러 게임 인터페이스** — 새 게임 = `lib/happy_trizn/games/<game_name>/` 폴더 하나 + 클라이언트 자산.
6. **일괄 7게임 출시** — 단계 출시 비추 권고했지만 사용자 일괄 결정. 모듈 인터페이스로 병렬 개발.
7. **확장 야망(디스코드/git/SVN 연동)은 무시** — 사용자 본인이 무시 지시.

## Approaches Considered

### Approach A: Elixir + Phoenix 모놀리식 (선택)

Phoenix LiveView (로비/채팅/친구) + Phoenix Channels (게임 실시간) + Ecto/MySQL + mongodb-elixir + 클라이언트 JS Canvas (게임 렌더링). 단일 Docker 컨테이너 + MySQL + MongoDB 컴포즈.

**Pros**: 실시간 끝판왕, BEAM 동시성, 색다름+유망 스택, 게임 방마다 supervisor process로 격리.
**Cons**: Elixir 학습 곡선, 게임 캔버스는 여전히 클라이언트 JS 작성 필요.

### Approach B: Go + Fiber + WebSocket

Go BE + Vanilla JS or htmx FE + sqlx/MySQL + mongo-go-driver.

**Pros**: 단순, 단일 바이너리, 학습 곡선 낮음.
**Cons**: 실시간 게임에 Phoenix만큼 우아하지 않음. 색다름 약함.

### Approach C: Rust + Tauri 데스크탑

Tauri + Axum BE.

**Pros**: 극도로 색다름, 네이티브 성능.
**Cons**: Tauri = 데스크탑 앱. "웹 열고 닉네임" 요구사항과 충돌. 사내 PC에 설치해야 함. → 부적합.

## Recommended Approach

**Elixir + Phoenix 모놀리식 + 클라이언트 JS Canvas 게임.**

### 데이터 분담

**MySQL (트랜잭션 영구 데이터):**
- `users` — id, email(`@trizn.kr` 락), nickname, password_hash, status(active/banned), created_at (admin은 DB 아닌 `.env`에 고정)
- `sessions` — id, user_id(nullable, 게스트 시 null), nickname, token_hash, expires_at, created_at (재배포 시 로그인 유지용)
- `friendships` — user_id, friend_id, status(pending/accepted), created_at (양방향 수락 모델)
- `rooms` — id, game_type, name, password_hash(nullable, sha256+salt), host_id, max_players, created_at
- `personal_records` — user_id, game_type, score, achieved_at (싱글게임 최고기록)
- `match_results` — id, game_type, room_id, winner_id, finished_at (멀티 등수 영구화)
- `admin_actions` — id, action(ban/unban/nickname_change/room_kill), target_user_id(nullable), target_room_id(nullable), payload(JSON), created_at (감사 로그)

**MongoDB (이벤트/타임시리즈/유연 스키마):**
- `game_events` — 게임 한 판의 모든 이벤트 (테트리스 블록 위치, Bomberman 폭탄 등). 리플레이용.
- `chat_logs` — 채널/DM 메시지 (글로벌 + DM 모두 영구). 인덱스: `{room: 1, ts: -1}`, `{user_a: 1, user_b: 1, ts: -1}`.
- `presence_logs` — 접속/이탈 이력
- `analytics` — 게임 플레이 시간, 추후 사내 도구 기능 추가 시 활용

날짜 형식: MySQL DATETIME `YYYY-MM-DD HH:MM:SS`, MongoDB BSON ISODate (UTC).

### 모듈러 게임 인터페이스

```elixir
# lib/happy_trizn/games/game_behaviour.ex
defmodule HappyTrizn.Games.GameBehaviour do
  @callback init(config :: map()) :: {:ok, state :: map()}
  @callback handle_input(player_id :: String.t(), input :: map(), state :: map()) ::
              {:ok, new_state :: map(), broadcast :: list()}
  @callback handle_player_join(player_id :: String.t(), meta :: map(), state :: map()) ::
              {:ok, new_state :: map(), broadcast :: list()} | {:reject, reason :: atom()}
  @callback handle_player_leave(player_id :: String.t(), reason :: :quit | :disconnect | :kick, state :: map()) ::
              {:ok, new_state :: map(), broadcast :: list()}
  @callback tick(state :: map()) :: {:ok, new_state :: map(), broadcast :: list()}
  @callback game_over?(state :: map()) :: {:yes, results :: map()} | :no
  @callback terminate(reason :: term(), state :: map()) :: :ok
  @callback meta() :: %{name: String.t(), max_players: integer(), mode: :multi | :single}
end
```

각 게임 = `lib/happy_trizn/games/tetris/`, `bomberman/` 등 폴더. GenServer가 방마다 한 개 spawn. 죽어도 Supervisor가 재시작 (한 방 죽어도 다른 방 영향 없음).

클라이언트 측도 동일 패턴: `assets/js/games/<game>/index.js`가 `IGameClient` 인터페이스 구현 (mount, render, sendInput).

### 인증 플로우

```
[게스트]
브라우저 → 닉네임 입력 → Phoenix.Token 임시 세션 → 입장
- 멀티 게임 OK, 채팅 OK, 친구 추가 안 됨
- 점수/등수 기록 휘발 (세션 종료 시 사라짐)

[등록자]
회원가입 → @trizn.kr 메일 입력 + 비번 → bcrypt 해시 저장, status=active 즉시
로그인 → bcrypt 검증 → 세션 토큰
- 모든 기능, 영구 기록, 친구 추가/수락
- 메일 인증 없음. 도메인 정규식 락만으로 외부인 차단.
- 의심스러운 가입은 admin이 사후 ban.
```

이메일 도메인 검증:
```elixir
def changeset(user, attrs) do
  user
  |> cast(attrs, [:email, :nickname, :password])
  |> validate_format(:email, ~r/^[^\s]+@trizn\.kr$/i)
  |> validate_length(:password, min: 8)
  |> unique_constraint(:email)
  |> unique_constraint(:nickname)
end
```

메일 발송 인프라 불필요. SMTP 라이브러리 의존성 제거.

### Admin 페이지

`/admin` 경로 (role=admin만 접근, Plug로 가드):

- **사용자 목록** — 전체 users 테이블 조회. 컬럼: id, email, nickname, status, 가입일, 마지막 접속, 게임 플레이 횟수
- **검색/필터** — 닉네임/이메일/status별 필터
- **액션** — ban/unban, 닉네임 강제 변경
- **방 모니터** — 현재 활성 방 리스트 (Phoenix Presence), 강제 종료
- **신고/로그** — chat_logs Mongo 조회 (욕설/스팸 검토용)
- **통계** — 일/주/월 활성 사용자, 게임별 플레이 카운트, 인기 시간대

### Admin 인증 (고정 계정)

**users 테이블에 admin 계정 없음.** 일반 사용자와 분리. `.env`에서 ID/비번 고정 주입:

```bash
# .env
ADMIN_ID=admin
ADMIN_PASSWORD_HASH=$2b$12$...   # bcrypt hash (운영 시 평문 금지)
ADMIN_SESSION_SECRET=<32바이트 랜덤>
```

```elixir
# config/runtime.exs
config :happy_trizn, :admin,
  id: System.fetch_env!("ADMIN_ID"),
  password_hash: System.fetch_env!("ADMIN_PASSWORD_HASH")
```

`/admin/login` 라우트:
- POST id/password 받음 → `Bcrypt.verify_pass/2`로 검증
- 성공 시 별도 세션 키(`:admin_authenticated`) 세팅, TTL 짧게(2시간)
- Plug `EnsureAdmin`이 모든 `/admin/*` 라우트 가드

회전: `.env` 수정 후 컨테이너 재시작. DB 마이그레이션 불필요.
보안: HTTPS 필수, 세션 쿠키 `secure + httponly + sameSite=Strict`, 로그인 실패 rate limit (5회/분).

### 채팅 + 친구

- **글로벌 채팅** Phoenix Channel `lobby:main`
- **DM** Channel `dm:#{user_a_id}-#{user_b_id}` (등록자만)
- **친구 목록 사이드바** LiveView, 모든 사용자 표시 + "추천 친구 10명 + 더보기" 버튼
- **친구 추가** = `friendships` row insert(`status=pending`) → 상대방 알림(LiveView 브로드캐스트) → 상대 수락 시 `accepted`

### 방 시스템 (멀티게임)

- **방 생성** — 게임 타입 선택 → name/password/max_players 입력 → MySQL `rooms` insert + Phoenix Presence 추적
- **방 참여** — 비밀번호 검증 → Channel join → GenServer state에 player 등록
- **강퇴** — 호스트만 가능, Channel push로 강제 leave + ban 짧게(5분)
- **방 리스트** — LiveView 실시간 갱신 (Presence diff 구독)

### Docker 배포

**포트 충돌 회피.** 사용자 개발 서버에 다른 프로젝트들 이미 표준 포트(4000/3306/27017) 점유 중일 수 있음. 호스트 노출 포트는 **비표준 + 프로젝트 prefix `47xxx`**로 통일 (47 = "트리즌" 47년차 농담 핑계). 컨테이너 내부 포트는 그대로.

| 서비스 | 컨테이너 내부 | 호스트 노출 | 비고 |
|---|---|---|---|
| Phoenix app | 4000 (`PORT`) | **4747** (`PHX_PORT_HOST`) | UFW로 사내 서브넷만 허용 |
| MySQL | **4406** (`MYSQL_PORT`) | **(노출 안함)** | Docker 네트워크 내부만, mysql:8.0 |
| MongoDB | **37017** (`MONGO_PORT`) | **(노출 안함)** | Docker 네트워크 내부만 |
| MySQL (dev override) | 4406 | **47306** | `docker-compose.override.yml`에서만 |
| MongoDB (dev override) | 37017 | **47017** | `docker-compose.override.yml`에서만 |

**모든 포트는 `.env` 단일 소스.** docker-compose 가 `${VAR:-기본값}` 패턴으로 fallback. `.env` 한 곳만 바꾸면 모든 서비스 추종.

`docker-compose.yml` (운영, 4 서비스):
```yaml
services:
  app:
    build: { context: ., dockerfile: Dockerfile }
    ports: ["${PHX_PORT_HOST:-4747}:${PORT:-4000}"]
    env_file: .env
    environment: { PHX_SERVER: "true" }
    depends_on:
      migrate: { condition: service_completed_successfully }
      mongo:   { condition: service_healthy }

  # oneshot — mysql healthy 후 마이그 실행 → exit. up -d 한 번에 자동 마이그.
  migrate:
    image: happy_trizn:latest
    restart: "no"
    env_file: .env
    environment: { PHX_SERVER: "false" }
    command: ["/app/bin/migrate"]
    depends_on:
      mysql: { condition: service_healthy }

  mysql:
    image: mysql:8.0
    command: ["--port=${MYSQL_PORT:-4406}"]
    expose: ["${MYSQL_PORT:-4406}"]
    environment:
      MYSQL_DATABASE: "${MYSQL_DATABASE}"
      MYSQL_ROOT_PASSWORD: "${MYSQL_ROOT_PASSWORD}"
    volumes: [mysql_data:/var/lib/mysql]
    healthcheck: { ... }

  mongo:
    image: mongo:7
    command: ["--port", "${MONGO_PORT:-37017}"]
    expose: ["${MONGO_PORT:-37017}"]
    volumes: [mongo_data:/data/db]
    healthcheck: { ... }

networks:
  happy_net: { name: happy_trizn_net }
volumes:
  mysql_data: { name: happy_trizn_mysql_data }
  mongo_data: { name: happy_trizn_mongo_data }
```

`docker-compose.override.yml` (개발 디버깅, gitignored):
```yaml
services:
  mysql: { ports: ["47306:4406"] }
  mongo: { ports: ["47017:37017"] }
```

UFW 룰:
```bash
sudo ufw allow from <사내_서브넷> to any port 4747 proto tcp
sudo ufw deny 4747
# DB 포트는 host 노출 X라 UFW 룰 불필요
```

`.env` (gitignored, 인라인 주석 미지원 — KEY=VALUE 한 줄만):
```
PHX_HOST=happy.trizn.local
PORT=4000
PHX_PORT_HOST=4747
SECRET_KEY_BASE=...
LIVE_VIEW_SIGNING_SALT=...
MYSQL_HOST=mysql
MYSQL_PORT=4406
MYSQL_DATABASE=happy_trizn
MYSQL_USER=root
MYSQL_PASSWORD=<강력한 비번>
MYSQL_ROOT_PASSWORD=<같은 값 OK>
MONGO_PORT=37017
MONGO_URL=mongodb://mongo:37017/happy_trizn
ADMIN_ID=trizadmin
ADMIN_PASSWORD_HASH=$2b$12$...
ADMIN_SESSION_SECRET=...
```

DATABASE_URL 미사용 — 비번에 URL reserved char(`&%^`) 들어가도 안전. runtime.exs 가 `MYSQL_HOST` 있으면 hostname/port/user/password 직접 사용.

**볼륨/네트워크 이름도 prefix.** 다른 프로젝트의 동명 볼륨/네트워크와 충돌 방지: `happy_trizn_mysql_data`, `happy_trizn_mongo_data`, `happy_trizn_net`.

Dockerfile = multi-stage Elixir release build. 최종 이미지는 OTP release만 (~50MB).

## Resolved Decisions (eng-review 2026-04-26)

- **Tetris 권위 모델**: 클라이언트 권위 (Jstris 모델). 서버는 가비지/점수/승패만 동기화.
- **GameBehaviour 콜백**: `handle_player_join`, `handle_player_leave(reason)`, `terminate` 추가.
- **재배포 정책**: 게임방 ETS = wipe 인정 (사용자에게 사전 안내 배너). 로그인 세션은 `sessions` 테이블 + cookie token으로 영속.
- **친구 모델**: 양방향 수락 (`pending → accepted`).
- **Admin**: `.env` 단일 고정 계정. 감사 로그는 `admin_actions` 테이블로 보완.
- **Chat 보존**: 글로벌 + DM 모두 영구 (Mongo `chat_logs`).
- **방 비번 해시**: sha256 + salt (bcrypt over-kill, 임시 인증용).
- **Channel 재연결**: 클라이언트 재연결 시 GenServer가 `catch_up` 메시지로 현재 state 스냅샷 push.
- **MongoDB fallback (Sprint 4)**: `Broadway` 파이프라인으로 game_events 비동기 큐잉. Mongo 다운 시 in-memory buffer 후 retry. 게임 진행 막지 않음.
- **호스트 disconnect 정책** (`handle_player_leave/3`):
  - Tetris 1v1: 남은 1명 자동 승리, GenServer terminate
  - Bomberman/Snake.io: 다음 join 순서 사람에게 호스트 권한 이전. 모두 떠나면 terminate.
  - Skribbl: 그리는 사람이면 다음 차례 즉시 진행. 일반 참여자면 단순 leave. 방장 권한은 다음 join 순서로 이전.
  - 싱글 게임(2048/Minesweeper/Pac-Man): N/A (혼자 플레이)

### 추가 (Sprint 1 dev 부팅 검증 + 테스트 작성 중 발견)

- **컨테이너 내부 포트도 비표준** (4406 / 37017). 호스트 노출만 47xxx 가 아니라, 다른 docker-compose 프로젝트와 같은 docker network 안에서 충돌 회피.
- **`mysql:8.0` 명시**: `mysql:8` (latest) 은 8.4+ 로 가서 `--default-authentication-plugin` 옵션 거부. 8.0 으로 lock.
- **`PHX_SERVER=true` env 분기**: 같은 이미지를 oneshot migrate 와 운영 app 모두에서 재사용. migrate 는 `false` 로 띄워서 Phoenix endpoint 안 시작.
- **`oneshot migrate` service**: `up -d` 시 자동 마이그. `service_completed_successfully` condition 으로 app 이 기다림.
- **WebSocket origin 화이트리스트**: prod 모드 endpoint 가 `PHX_HOST` 만 허용 → 브라우저 `localhost:4747` 거부. `runtime.exs` 에서 `check_origin: ["//#{host}", "//localhost", "//127.0.0.1", ...]` + `CHECK_ORIGIN` env override.
- **`DATABASE_URL` 미사용**: 비번에 URL reserved char(`&%^`) 들어가도 안전하게 `MYSQL_HOST/PORT/USER/PASSWORD` 직접 env 로 전달.
- **`Phoenix.Token.sign/verify` 인자**: binary (20+ chars) 직접 전달. 기존 `%{secret_key_base: ...}` map 패턴 매칭 실패 (테스트 작성 중 발견).
- **`mix compile` 먼저, 그 다음 `mix assets.deploy`**: Phoenix LiveView 1.1 의 `phoenix-colocated/<app>` 디렉토리는 컴파일 시 자동 생성. esbuild 가 import 하므로 컴파일 후에 assets.deploy.
- **`bin/mix` wrapper docker network 자동 join**: `happy_trizn_net` 떠있으면 합류 + `--env-file .env` → `mix test` 가 mysql 호스트네임 그대로 접근.
- **`.env` 인라인 주석 금지**: docker `--env-file` 미지원. 주석은 별도 줄 `#` 으로.
